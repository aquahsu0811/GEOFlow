# Report Completeness Model

GEO Signal Monitor 报告必须先系统、详细、完整，再追求表达简洁。除非用户明确要求短版，每份正式报告都按 `S-D-C` 三层自检。

## S-D-C 自检

| 层级 | 要求 | 检查问题 |
|---|---|---|
| Systematic | 结构覆盖完整链路 | 是否覆盖来源、场景、Prompt、采样、指标、引用、归因、纠偏、看板、治理和附录？ |
| Detailed | 每个模块有可执行字段 | 是否有样本字段、阈值、负责人、验收指标、时间窗口、来源 URL 和置信度？ |
| Complete | 结论可审计、可复采 | 是否能从结论回到样本、Prompt、平台、引用和来源账本？ |

## 六层分析模型

| 层 | 问题 | 必备输出 |
|---|---|---|
| 可见性 | 品牌是否出现、入选、被推荐、排序如何 | 出现率、候选率、推荐率、平均排序、平台差异 |
| 事实性 | 答案是否准确描述品牌、产品、价格、客户、资质 | 描述准确率、事实错误表、事实卡、错误等级 |
| 证据性 | 答案引用了什么，引用是否支持说法 | 引用召回率、引用准确率、来源类型、支持等级 |
| 稳定性 | 同一问题多次答案是否波动 | 样本数、波动区间、结论稳定性、引用稳定性 |
| 竞争性 | 竞品如何出现，是否压制品牌 | 竞品出现率、替代关系、比较维度、优势/弱项 |
| 治理与归因 | 变化是否可解释，风险是否可控 | 基线/观察窗口、对照 Prompt、混杂因素、合规边界 |

## 正式报告模块

1. 执行摘要与关键判断
2. 任务范围、边界和采样声明
3. 权威参考与来源账本
4. 公司/品牌事实基线
5. 公司测试场景发现
6. 监测 Prompt 库与对照组
7. 真实数据接入模式与证据等级
8. 五平台采样口径
9. 核心指标总览
10. 平台差异分析
11. 引用源追踪与证据质量
12. 答案事实性与描述准确率
13. 竞品、替代和负面表述分析
14. 稳定性、波动和置信度
15. 答案差异与谨慎归因
16. 纠偏任务与内容/页面/信源路线图
17. 告警规则和复盘节奏
18. 仪表盘字段、数据库表和 API 草案
19. 治理、合规、数据质量和风险控制
20. 自 review 结果
21. 附录：Prompt 全表、指标字典、来源账本、采样字段

## 来源账本字段

每条关键来源至少记录：

- `source_id`
- `source_type`
- `title`
- `url`
- `publisher`
- `retrieved_at`
- `fact_supported`
- `freshness_risk`
- `how_used`
- `confidence`

## HTML 可视化要求

- HTML 必须有可跳转目录菜单。
- 桌面端目录使用 `position: fixed`，页面下拉时固定跟随。
- 移动端目录改为顶部 `position: sticky`，避免遮挡正文。
- 打印和 PDF 输出中隐藏固定目录，避免重复覆盖页面内容。
- 表格必须有边框、可断行、无横向溢出。
