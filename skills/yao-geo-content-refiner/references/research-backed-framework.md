# 研究支撑框架

本 skill 的理论基础是把旧内容改造成更容易被生成式引擎检索、理解、引用、核验和复用的知识资产。它不把 GEO 简化为关键词堆叠，而是把内容拆成“可回答、可检索、可证明、可抽取、可发布、可复核”的结构。

## 权威参考基线

| 参考方向 | 参考对象 | 对本 skill 的约束 |
|---|---|---|
| GEO | `GEO: Generative Engine Optimization`，KDD 2024 / arXiv:2311.09735 | 关注生成式引擎回答中的内容可见性；不同领域策略效果不同，所以报告必须保留行业、选题和平台差异。 |
| RAG | `Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks`，NeurIPS 2020 / arXiv:2005.11401 | 内容要能被检索、拼接和归因；必须输出稳定标题、定义段、事实段、表格和来源锚点。 |
| 带引用生成 | `Enabling Large Language Models to Generate Text with Citations`，ALCE / arXiv:2305.14627 | 答案、证据、引用质量要分开检查；每条事实必须能映射到来源或进入缺口。 |
| 长上下文鲁棒性 | `Lost in the Middle`，TACL / arXiv:2307.03172 | 核心结论、适用边界和证据摘要必须前置，避免关键信息只埋在长文中段。 |
| 原子事实核验 | `FActScore`，EMNLP 2023 / arXiv:2305.14251 | 长文结论要拆成原子事实卡，逐条判断来源支持，不用二元“真/假”粗略评价。 |
| 自反思式生成 | `Self-RAG`，arXiv:2310.11511 | 报告必须包含自 Review，检查是否需要补检索、补证据、降级结论或修复排版。 |
| 内容可信度 | Google Search Quality Rater Guidelines 2025 | 对作者/来源/更新时间/声誉/主内容质量做独立检查，敏感行业按更高证据门槛处理。 |

## 设计成能力

| 能力层 | 具体要求 | 输出落点 |
|---|---|---|
| 问题层 | 明确主问题、追问、反向问法、平台问法和购买/决策意图。 | FAQ、平台适配矩阵、语义与实体地图 |
| 结构层 | 直接回答前置，正文按 H2/H3、步骤、表格、FAQ、来源列表组织。 | GEO 改造版文章、CMS HTML 建议 |
| 事实层 | 将产品能力、数据、时间、范围、限制拆成原子事实。 | 原子事实卡、证据强度与缺口 |
| 证据层 | 区分官方来源、用户资料、第三方来源、待核验资料和证据不足。 | 证据强度与缺口、差异报告、自 Review |
| 数据层 | 区分公开可核验、用户提供、授权连接、不可访问和需脱敏数据。 | 真实数据获取与核验计划、来源访问记录 |
| 语义层 | 补实体全称、别名、术语、上下位关系、场景词和查询重写词。 | 语义与实体地图、FAQ |
| 平台层 | 按国内 AI 平台和微信生态的共同特征改造，而不是只做传统 SEO。 | 平台适配矩阵、页面发布建议 |
| 发布层 | 给 CMS、公众号、官网、知识库和监测动作留出落地建议。 | 发布与追踪建议、效果观察计划 |

## 不应误用

- 不把 GEO 论文中的可见性提升数字写成具体客户承诺。
- 不把 RAG 研究等同于“只要加来源就一定会被引用”。
- 不把官方营销页中的效果类文案直接升级为因果结论。
- 不用统一模板抹平行业差异；医疗、金融、法律、教育等场景必须提高证据门槛和合规过滤。
