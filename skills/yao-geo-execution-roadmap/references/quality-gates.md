# Quality Gates

## 内容自检

- 是否承接诊断数据，而不是通用建议。
- 是否说明真实数据能力边界、获取方式、来源权限、新鲜度和待补证缺口。
- 是否包含十维分析完整性矩阵，并写清当前状态、缺口、动作、验收和责任角色。
- 是否包含证据成熟度与事实底座，区分 A/B/C/D/E 证据等级和禁用表述。
- 是否包含六个项目包。
- 每个项目包是否有目标、任务、输入、步骤、负责人、交付物、验收指标、依赖关系、优先级和资源预算。
- 是否有 30/60/90 天路线图。
- 是否包含资源预算与优先级、治理节奏与决策机制。
- 是否覆盖 DeepSeek、豆包、千问、Kimi、元宝。
- 是否区分“提升可发现性/可验证性/可抽取性”与“保证平台引用”。
- 是否包含风险预案和敏感行业合规节点。

## 文件自检

- Markdown、HTML、Word、PDF 四个文件真实存在且非空。
- HTML 背景为白色，表格边框对齐，长词不溢出。
- HTML 是否使用 kami 风格的暖白辅助面、油墨蓝强调、serif 标题、sans 正文和紧凑行距。
- HTML 必须有 `report-menu` 固定菜单、章节锚点和 `position: sticky`。
- 来源台账必须包含来源类型、证据等级、核验日期和新鲜度。
- Word 文件是有效 docx zip，正文包含核心章节。
- Word 文件中的所有表格必须有固定 `tblW`、`tblGrid` 和 `tcW`，不得出现 `w:type="auto"` 表格宽度，所有列宽总和不得超过页面可用宽度。
- PDF 文件以 `%PDF` 开头，文件大小合理。
- `quality-report.json` 写明每一项检查结果。

## 失败处理

只要任何关键项失败，先修复内容或生成器，再重新生成四格式报告。不要交付缺文件、空文件或只包含标题的 Word/PDF。
